## 一、目标一：类图设计（初步方案）

### 1.1 核心领域类识别

基于 P1 需求分析与 P2 架构设计，CampusHub 系统的核心领域类如下：

| 类名 | 类型 | 职责简述 |
|------|------|---------|
| `User` | 抽象基类 | 用户通用属性与行为，含注册、登录、资料管理 |
| `Requester` | 子类（继承 User） | 需求方：发布需求、查看订单、提交评价 |
| `Provider` | 子类（继承 User） | 服务方：浏览需求、接单、完成订单、提交评价 |
| `Admin` | 子类（继承 User） | 管理员：用户管理、内容审核、数据统计 |
| `Task` | 实体类 | 互助需求：标题、分类、描述、报酬、状态、时限 |
| `Order` | 实体类 | 订单：关联需求方与服务方，跟踪执行状态 |
| `Review` | 实体类 | 双向评价：评分、评论、关联订单 |
| `Notification` | 实体类 | 消息通知：订单状态变更、新评价、系统通知 |
| `Category` | 枚举/实体 | 需求分类：快递代取、学习辅导、二手交易、组队匹配 |

**关系说明：**

| 关系 | 说明 |
|------|------|
| User → Requester / Provider / Admin | **继承**：子类继承 User 通用属性，各自扩展特定行为 |
| Task → Category | **关联**：每个 Task 属于一个 Category |
| Requester (1) → Task (*) | **一对多**：一个需求方发布多个需求 |
| Provider (1) → Order (*) | **一对多**：一个服务方接多个订单 |
| Task (1) → Order (1) | **一对一**：一个需求对应一个有效订单（核心约束） |
| Order (1) → Review (1..2) | **一对多**：一个订单产生 1-2 条评价（双方互评） |
| User (1) → Notification (*) | **一对多**：一个用户有多条通知 |

### 1.3 关键属性与方法（初步）

**User（抽象类）：**
- 属性：`id: Long`, `name: String`, `email: String`, `phone: String`, `passwordHash: String`, `avatar: String`, `creditScore: Double`, `createdAt: DateTime`
- 方法：`login(email, password): User`, `updateProfile(data): void`

**Requester：**
- 扩展属性：`publishedTaskCount: Integer`, `completionRate: Double`
- 方法：`publishTask(taskData): Task`, `viewMyTasks(): List<Task>`

**Provider：**
- 扩展属性：`completedOrderCount: Integer`, `avgRating: Double`
- 方法：`acceptOrder(taskId): Order`, `viewAcceptedOrders(): List<Order>`

**Task：**
- 属性：`id: Long`, `title: String`, `description: String`, `category: Category`, `reward: BigDecimal`, `status: TaskStatus`, `location: String`, `deadline: DateTime`, `requesterId: Long`, `createdAt: DateTime`
- 方法：`publish(): void`, `close(): void`

**Order：**
- 属性：`id: Long`, `taskId: Long`, `requesterId: Long`, `providerId: Long`, `status: OrderStatus`, `createdAt: DateTime`, `completedAt: DateTime`
- 方法：`accept(): void`, `complete(): void`, `cancel(): void`

**Review：**
- 属性：`id: Long`, `orderId: Long`, `reviewerId: Long`, `targetId: Long`, `score: Integer (1-5)`, `comment: String`, `createdAt: DateTime`
- 方法：`submit(): void`

**Notification：**
- 属性：`id: Long`, `receiverId: Long`, `type: NotifyType`, `content: String`, `isRead: Boolean`, `createdAt: DateTime`

### 1.4 SOLID 原则预检

| SOLID 原则 | 设计决策 | 状态 |
|-----------|---------|------|
| **S - 单一职责** | 将用户按角色拆分为 Requester/Provider/Admin，每类只负责自身角色的行为。Order 只管理订单状态流转，不处理评价逻辑。 | 已考虑 |
| **O - 开闭原则** | Task 的状态流转通过状态模式实现，新增状态无需修改 Task 类。Category 作为可扩展枚举，新增分类不影响现有逻辑。 | 已考虑 |
| **L - 里氏替换** | Requester、Provider、Admin 均可替换 User 使用，不改变程序正确性。所有子类遵循基类契约。 | 已考虑 |
| **I - 接口隔离** | 将通知接口拆分为 `INotificationSender`（发送）和 `INotificationReceiver`（接收），避免"胖接口"。 | 已考虑 |
| **D - 依赖倒转** | Service 层依赖 `INotificationService` 接口而非具体实现，方便切换通知方式（站内信/推送）。 | 已考虑 |

> **注：** 后续将按 P3 要求，先让 AI 生成一版类图，再用 SOLID 清单逐条审查 AI 的设计，记录违反情况并修正。

### 1.5 设计模式识别

#### 模式一：状态模式（State Pattern）— Order 状态管理

- **应用位置：** `Order` 的状态流转
- **为什么用：** Order 有 `待接单 → 已接单 → 进行中 → 已完成 / 已取消` 等多个状态，每个状态下的合法操作不同。状态模式将状态行为封装到独立的状态类中，避免 Order 中出现大量 if-else 分支。
- **不用会怎样：** Order 类中将充斥状态判断逻辑（`if (status == PENDING) ... else if ...`），每新增一种状态需修改多处代码，违反开闭原则。

#### 模式二：策略模式（Strategy Pattern）— 任务排序/推荐策略

- **应用位置：** Task 列表的排序和推荐逻辑
- **为什么用：** 首页需求列表可能需要多种排序策略（按时间、按报酬、按距离、按推荐权重），策略模式允许运行时切换排序算法，且未来可扩展新的推荐算法（如基于用户画像的智能推荐）。
- **不用会怎样：** 排序逻辑硬编码在 Service 层，新增排序方式需修改现有方法，维护困难，且无法在运行时动态切换。

#### 模式三（可选）：观察者模式（Observer Pattern）— 通知系统

- **应用位置：** 订单状态变更时的通知推送
- **为什么用：** 当 Order 状态变更时，需通知需求方和服务方。观察者模式解耦订单逻辑与通知逻辑，Order 只负责发布状态变更事件，通知模块订阅并处理。
- **不用会怎样：** Order 类需要直接调用 NotificationService，增加不必要的耦合，后续若需增加短信/邮件通知会更困难。

## 二、目标二：API 接口设计（初步方案）

### 2.1 API 总体规范

| 项目 | 规范 |
|------|------|
| 风格 | RESTful |
| Base URL | `https://api.campushub.example.com/v1` |
| 认证方式 | JWT Bearer Token（登录后颁发，过期时间 24h） |
| 数据格式 | JSON（请求/响应均使用 `application/json`） |
| 字符编码 | UTF-8 |
| 版本策略 | URL 路径版本 `/v1/` |

### 2.2 统一响应格式

**成功响应：**
```json
{
  "code": 200,
  "message": "success",
  "data": { ... }
}
```

**分页响应：**
```json
{
  "code": 200,
  "data": {
    "list": [ ... ],
    "page": 1,
    "pageSize": 20,
    "total": 100
  }
}
```

**错误响应：**
```json
{
  "code": 40001,
  "message": "参数校验失败：标题不能为空",
  "data": null
}
```

### 2.3 核心接口详细定义

---

#### 接口 1：用户注册 `POST /v1/auth/register`

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `name` | string | ✅ | 真实姓名，2-20 字符 |
| `email` | string | ✅ | 学校邮箱，需 `.edu` 结尾 |
| `password` | string | ✅ | 密码，8-32 字符，含大小写字母+数字 |
| `phone` | string | ✅ | 手机号，11 位 |
| `role` | string | ✅ | 角色：`requester` / `provider` |

**成功响应 (201)：**
```json
{
  "code": 201,
  "message": "注册成功",
  "data": {
    "userId": 1001,
    "name": "张三",
    "role": "requester"
  }
}
```

**错误示例：**
| 错误码 | 说明 |
|--------|------|
| `40001` | 参数校验失败（如邮箱格式不正确） |
| `40002` | 邮箱已被注册 |
| `40003` | 密码强度不足 |

### 2.4 错误码体系

| 错误码范围 | 类别 | 示例 |
|-----------|------|------|
| `40001-40099` | 参数校验 / 业务规则 | 参数为空、重复操作、状态不允许 |
| `40100-40199` | 认证相关 | 登录失败、Token 过期、未登录 |
| `40300-40399` | 权限相关 | 角色不允许、无权访问资源 |
| `40400-40499` | 资源不存在 | 订单不存在、用户不存在 |
| `50000-50099` | 服务器内部错误 | 数据库异常、未知错误 |

### 2.5 安全性要点（初步）

| 要点 | 措施 |
|------|------|
| 密码存储 | bcrypt 加盐哈希，永不存储明文 |
| 手机号保护 | 数据库中 AES 加密存储，API 返回时脱敏（`138****1234`） |
| 认证 | 所有非公开接口需 JWT Token，过期 24h |
| 防越权 | 查询订单/提交评价时校验用户是否为订单参与方 |
| 输入校验 | 所有用户输入在后端做二次校验，防范 XSS/SQL 注入 |
