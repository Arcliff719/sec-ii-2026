# 任务 3 讲解：数据库设计思路

---

## 一、从需求到 ER 图 —— 实体是怎么"筛"出来的

数据库设计的第一步，不是画图也不是写 SQL，而是**从需求文档里"筛"出实体**。

### 1.1 实体的来源

回头看 P1 的需求规格说明书，CampusHub 的核心业务流程是：

> 用户注册 → 发布需求 → 有人接单/参与 → 完成 → 服务方评价需求方

把这段话拆开，里面每一个「名词」都是一个潜在的实体候选：

| 业务描述 | 候选实体 | 是否保留？ |
|---------|---------|-----------|
| **用户**注册 | User | ✅ 核心 |
| 发布**需求** | Task | ✅ 核心 |
| 需求有**分类** | Category | ✅ Task 依赖它，且分类决定是否可评价 |
| 有人**接单**/参与 | Order | ✅ 核心（统一承载所有用户-Task 连接） |
| 服务方评**价** | Review | ✅ 核心（只对 Order 负责） |
| 状态变更通知 | Notification | ✅ 辅助 |

### 1.2 为什么不把 Requester/Provider/Admin 拆成三张表？

P3 类图设计中，Requester、Provider、Admin 是 User 的子类（继承关系）。但数据库设计中，**继承关系有三种映射策略**：

| 策略 | 做法 | 适用场景 |
|------|------|---------|
| 单表继承 | 一张 `users` 表，用 `role` 字段区分 | 子类属性差异小 |
| 具体表继承 | 三张独立的表 | 子类差异大 |
| 类表继承 | 基表 + 各子类补充表 | 共享字段多但又有特有字段 |

CampusHub 选择**单表继承**，理由：三种角色共有属性远多于差异属性，且登录不区分角色，单表查询最快。

## 二、关系怎么定 —— 拆解每个"动词"

### 2.1 Task → Order 为什么是 1:N？

第一版设计中，Task → Order 是 1:1——一个需求只能被一个人接。但仔细想"组队匹配"：

> 用户 A 发帖"周末篮球 4v4，缺 3 人"，B、C、D 先后加入

这里一个 Task 需要关联多个用户。如果强制 1:1，组队匹配就没法用同一套表结构。

**最终方案：Task → Order = 1:N。** `orders` 表的 `idx_task_id` 是普通索引而非唯一索引。至于一个 Task 到底允许多少个 Order，由应用层根据分类决定：

- 快递代取 / 学习辅导 / 二手交易：应用层在创建 Order 前检查该 Task 是否已有活跃 Order，有则拒绝
- 组队匹配：应用层不做此限制，允许多人参与

这样 **一张 `orders` 表统一承载所有用户-Task 连接**，不因为交互模式的差异而建两套表。

### 2.2 Category 为什么只留 `has_review` 一个开关？

最初设计中 Category 有 `interaction_type`（one_to_one / many_to_many）和 `has_review` 两个字段。后来发现 `interaction_type` 本质上只影响一件事：**Task → Order 是 1:1 还是 1:N**。而这个约束放在应用层更灵活——不需要为两种模式建两张表。

`has_review` 则是必须保留的，因为它直接决定了业务流程的分叉：

```
订单完成后：
  → Category.has_review = 1 → 服务方可以评价需求方 → Review 表写入
  → Category.has_review = 0 → 跳过评价流程
```

| 分类 | has_review | 效果 |
|------|:---:|------|
| 快递代取 | 1 | 订单完成 → 评需求方 |
| 学习辅导 | 1 | 订单完成 → 评需求方 |
| 二手交易 | 0 | 无评价 |
| 组队匹配 | 0 | 无评价 |

**新增分类时只需指定 has_review，无需改任何代码。**

### 2.3 Order → Review 为什么是 1:0..1？Review 为什么不直接关联 User？

Review 表只存 `order_id`。需要知道谁评了谁时：

```sql
SELECT r.*, o.provider_id AS reviewer_id, o.requester_id AS target_id
FROM reviews r
JOIN orders o ON r.order_id = o.id;
```

好处：
- 消除冗余——provider/requester 只在 Order 中存储（单一真相源）
- 不可能出现"评价者不是该订单服务方"的数据异常
- UNIQUE KEY 在 order_id 上，自然保证一个订单一条评价

### 2.4 Notification 为什么不和业务表强绑定？

通知本质是"消息日志"，只存 `receiver_id` 和通知内容文本。不设外键到 Order/Review，避免：
- 业务数据被删后通知跟着丢失
- 历史通知中出现 NULL 的外键

## 三、索引设计 —— 哪些该建，哪些不建

> **只给 WHERE、JOIN、ORDER BY 里高频出现的列建索引。**

### 3.1 逐个分析

| 索引 | 对应的查询场景 | 不建会怎样？ |
|------|-------------|------------|
| `uk_email / uk_phone`（users） | 登录 `WHERE email = ?` | 唯一索引 = 查找快 + 不重复 |
| `idx_requester_id`（tasks） | "我的发布" | 全表扫描 |
| `idx_category_id`（tasks） | 首页分类筛选 | 全表扫描 |
| `idx_status`（tasks） | 只展示已发布的需求 | 低基数列但首页高频组合查询需要 |
| `idx_published_at`（tasks） | 首页按时间降序 | filesort，数据量大时很慢 |
| `idx_task_id`（orders） | 查某个 Task 的所有参与者 | 全表扫描 |
| `idx_requester_id`（orders） | 需求方"我的订单" | 全表扫描 |
| `idx_provider_id`（orders） | 服务方"我的接单" | 全表扫描 |
| `idx_status`（orders） | 按状态筛选 | — |
| `uk_order_id`（reviews） | 一个订单一条评价 | 可能重复评价 |
| `idx_receiver_read`（notifications） | 查某用户未读通知 | 复合索引 receiver_id 在前 |

### 3.2 `idx_task_id` 为什么是普通索引而不是唯一索引？

这是本次设计里刻意为之的。

如果设成 UNIQUE，组队匹配的"多人参与同一个 Task"就直接被数据库拦住了。设成普通索引，约束上移到应用层——需要 1:1 的分类在创建 Order 前自己检查，需要 1:N 的分类不做限制。**数据库层给灵活性，应用层加约束。**

## 四、SQL 类型选择

### 4.1 `credit_score` → DECIMAL(3,1)，不用 FLOAT

FLOAT 是近似存储，`0.1 + 0.2 ≠ 0.3`。信用分要参与排序和展示，小数点必须精确。

### 4.2 `status` → ENUM，不用 VARCHAR

| 对比 | ENUM | VARCHAR |
|------|------|--------|
| 存储 | 1-2 字节 | 实际长度 |
| 约束 | 自带校验 | 无 |
| 扩展 | ALTER TABLE | 直接写 |

状态值固定且有限，用 ENUM 合适。

### 4.3 `reward` → DECIMAL(10,2) DEFAULT NULL

报酬金额可以为空——组队匹配类任务不涉及金钱。应用层在前端按分类控制 reward 是否必填。

### 4.4 `phone` → VARCHAR(255) 而非 CHAR(11)

AES 加密后密文 + Base64 编码，长度远大于 11 且不固定，不能定长。

### 4.5 `is_read` → TINYINT(1)

MySQL 没有真正的 BOOLEAN，TINYINT(1) 就是最接近的类型。

## 五、隐私数据处理

### 5.1 密码：bcrypt

SHA256 是快哈希，GPU 每秒几十亿次。bcrypt 自带 salt + 可配置迭代轮数（cost=10 意味着 1024 轮），每秒只能试几次，暴力破解成本指数级上升。

### 5.2 手机号：AES 加密 + 展示脱敏

两层保护：
- 存储层：AES-256 密文，密钥存在环境变量不入库
- 展示层：API 返回 `138****1234`，即使合法用户也看不到别人完整号码

## 六、设计演进 —— 几个关键决策的来龙去脉

### 决策 1：Task → Order 从 1:1 改为 1:N

**起因：** 组队匹配需要多人参与同一个 Task，1:1 的 Order 模型无法容纳。

**曾考虑的方案：** 新建 `task_participants` 表专门处理组队匹配，与 `orders` 表并行。

**最终选择：** 删掉 `task_participants`，把 `orders.task_id` 的唯一索引改为普通索引，一张表统一承载。

**理由：** 两种表结构差异极小（字段几乎相同），多一套表意味着 DAO、Service、Controller 各多一套代码。应用层用十几行判断逻辑区分"一人接单"和"多人参与"，比维护两套表成本低得多。

### 决策 2：Review 不再直接关联 User

**起因：** 改为"只有服务方可以评价需求方"后，Review 表的 `reviewer_id` 和 `target_id` 分别等于 `Order.provider_id` 和 `Order.requester_id`——这是冗余。

**最终方案：** Review 只存 `order_id`，用户信息通过 JOIN 获取。

**理由：** 单一真相源。不可能出现"Review 说 reviewer 是用户 X，但 Order 说 provider 是用户 Y"这种数据异常。

### 决策 3：Category 只保留 has_review

**起因：** Category 曾有过 `interaction_type` 字段来区分 one_to_one 和 many_to_many。

**最终方案：** 只保留 `has_review`。交互模式的差异由应用层根据分类自行判断。

**理由：** `interaction_type` 只影响"允许多少个 Order"，而这个判断不需要数据库级别的约束——应用层在创建 Order 时查一下已有 Order 数就够了。`has_review` 则必须有，因为它直接影响"完成订单后要不要走评价流程"这条业务分支。

## 七、设计检查清单

- [ ] 每张表都有主键
- [ ] 所有外键列都建了索引
- [ ] 所有 WHERE 条件列都评估过是否需要索引
- [ ] 没有用 FLOAT/DOUBLE 存金额或分数
- [ ] 密码字段不存明文（bcrypt）
- [ ] 手机号字段有脱敏方案（AES 密文 + 展示脱敏）
- [ ] ENUM 的状态值覆盖了所有业务状态
- [ ] `has_review` 正确设定了默认值（默认 0 = 不评价）
- [ ] 建表顺序正确（users → categories → tasks → orders → reviews → notifications）
- [ ] 一人接单 + 多人参与两种场景都能走通同一套表

## 八、常见问题

**Q: 一个 Task 允许多个 Order，怎么防止快递代取被两个人同时接单？**

A: 应用层在创建 Order 前做检查——读 `Category` 的类型，如果需要 1:1，则查询该 Task 是否已有活跃状态的 Order（`status != cancelled`），有则拒绝。加上数据库事务的乐观锁或 `SELECT ... FOR UPDATE` 可以防止并发。

**Q: 为什么不给所有外键都加 ON DELETE CASCADE？**

A: 级联删除太危险。删一个 User 连带删掉所有 Task、Order、Review 无法恢复。更推荐"软删除"——加 `deleted_at` 字段。

**Q: `reward` 改成 DEFAULT NULL 后，怎么保证快递代取的报酬必填？**

A: 约束放在应用层/前端。发布 one_to_one 类型的 Task 时校验 reward 不为空，组队匹配则允许为空。

**Q: 新增分类时，除了 INSERT 一行还需要做什么？**

A: 如果新分类用现有交互模式（一人接单或多人参与），只需要 INSERT 时指定 `has_review`。如果引入全新的交互模式（如竞价），才需要加业务代码。大多数新增分类都不需要改代码。
