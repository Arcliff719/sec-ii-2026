# 需求发布板块 — 开发讲解

> 本文档汇总 P0-P4 各阶段中与"需求发布板块"相关的所有信息，帮助开发人员一次性理解需要做什么、需要哪些文件、涉及哪些技术决策。

---

## 一、板块概述

"需求发布板块"是 CampusHub 校园互助服务平台的 **P0 核心功能**之一，属于用户注册登录之后的第二步开发任务。

该板块包含三个子功能：

| 子功能 | 说明 | 难度 |
|--------|------|:---:|
| 发布需求 | 需求方填写表单发布互助需求 | ⭐⭐ |
| 需求列表 & 搜索 | 首页浏览、分类筛选、关键词搜索、分页 | ⭐⭐ |
| 需求详情 | 查看单个需求的完整信息 | ⭐ |

---

## 二、在架构中的位置

根据 P2 阶段确定的架构（ADR-1），项目采用 **分层架构 + 前后端分离**，该板块在四层架构中的位置：

```
┌─────────────────────────────────────────┐
│         展示层 (Vue 3 + Element Plus)     │
│  TaskPublish.vue / TaskList.vue /        │
│  TaskDetail.vue                          │
└────────────────┬────────────────────────┘
                 │ RESTful API (JSON)
                 │ JWT Token 认证
┌────────────────▼────────────────────────┐
│        Controller 层 (Spring MVC)        │
│  TaskController.java                     │
│  请求参数校验、响应格式化                  │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│         Service 层 (Spring @Service)      │
│  TaskService.java                        │
│  业务逻辑：角色校验、参数校验、分页查询     │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│       Repository 层 (Spring Data JPA)     │
│  TaskRepository.java                     │
│  CategoryRepository.java                 │
└────────────────┬────────────────────────┘
                 │
┌────────────────▼────────────────────────┐
│          数据库 (MySQL 8.0)               │
│  tasks 表 / categories 表 / users 表      │
└──────────────────────────────────────────┘
```

**分层纪律（P2 架构文档规定）：**
- Controller 层仅负责请求参数校验和响应格式化，不包含业务逻辑
- Service 层集中实现所有业务规则
- Repository 层仅负责数据存取
- **禁止跨层调用**

## 三、前后端归属

此板块属于 **前后端都涉及的全栈板块**。

### 前端负责：

| 功能 | 说明 |
|------|------|
| 发布表单 UI | 标题、描述、分类下拉、报酬、地点、截止时间 |
| 客户端校验 | 标题 5-50 字，描述 10-500 字，即时校验提示 |
| 需求列表页 | 列表展示 + 分类筛选栏 + 关键词搜索 + 分页 |
| 需求详情页 | 完整信息展示 |
| 角色权限控制 | 仅 requester 角色可见「发布需求」入口按钮 |
| 加载状态 & 错误提示 | Loading spinner, Toast 提示, 空状态页面 |

### 后端负责：

| 功能 | 说明 |
|------|------|
| `POST /v1/tasks` | 创建需求：校验参数 → 校验角色(仅requester) → 写入 tasks 表 |
| `GET /v1/tasks` | 分页查询：多条件筛选(分类/状态/关键词) + 排序 |
| `GET /v1/tasks/{id}` | 需求详情查询 |
| 参数二次校验 | 所有输入在后端再做校验（防绕过前端） |
| 角色校验 | 只有 requester 可发布 |
| 分类数据接口 | 提供分类列表供前端下拉选择 |

## 五、核心功能详细设计

### 5.1 发布需求 — `POST /v1/tasks`

**业务规则：**
- 仅 `requester` 角色可以发布
- 标题 5-50 字符，描述 10-500 字符
- 分类必须为 categories 表中存在的合法分类
- 截止时间不能早于当前时间
- 发布后需求状态默认为 `published`

**请求参数（来自 P3 API 规范）：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|:---:|------|
| title | string | 是 | 需求标题 |
| description | string | 是 | 需求描述 |
| category | string | 是 | 分类 |
| reward | integer | 否 | 报酬 |
| deadline | string | 是 | 截止时间 |
| location | string | 否 | 地点 |
| images | array | 否 | 图片URL列表 |

**前端校验规则：**
- 标题：5-50 字
- 描述：10-500 字
- 分类：必须从下拉列表选择（不能为空）
- 截止时间：必须晚于当前时间

**后端校验规则（防绕过前端）：**
- 所有字段二次校验
- 分类 ID 合法性校验
- 角色校验（从 JWT Token 中提取 role）
- 发布频率限制（可选：防刷）

### 5.2 需求列表 & 搜索 — `GET /v1/tasks`

**查询参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|:---:|------|
| category | string | 否 | 分类筛选 |
| status | string | 否 | 状态筛选 |
| page | integer | 否 | 页码（默认 1） |
| page_size | integer | 否 | 每页数量（默认 20） |
| sort | string | 否 | 排序方式（默认按发布时间倒序） |
| keyword | string | 否 | 关键词搜索（匹配标题） |

**响应结构（来自 P3 API 规范）：**

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "items": [
      {
        "demand_id": "dem_20240001",
        "title": "帮忙取快递",
        "description": "韵达快递，包裹比较大",
        "category": "delivery",
        "reward": 10,
        "status": "pending",
        "location": "3号宿舍楼下",
        "publisher": {
          "user_id": "123456",
          "nickname": "张三",
          "avatar": "https://..."
        },
        "created_at": "2024-01-15T10:30:00Z",
        "deadline": "2024-01-15T18:00:00Z"
      }
    ],
    "pagination": {
      "page": 1,
      "page_size": 20,
      "total": 45,
      "total_pages": 3
    }
  }
}
```

**数据库索引支持（P3 已设计）：**
- `idx_category_id` — 分类筛选
- `idx_status` — 状态筛选
- `idx_published_at` — 时间排序

### 5.3 需求详情 — `GET /v1/tasks/{id}`

**展示内容：**
- 需求基本信息（标题、描述、分类、报酬、地点、截止时间、状态）
- 发布者信息（昵称、头像、信用分）
- 发布时间
- 对接后续功能：详情页上的「接单」按钮（provider 角色可见，下一个板块开发）

## 六、技术栈对应

根据 P2 阶段的 ADR 决策：

| 层 | 选型 | ADR 依据 |
|----|------|---------|
| 前端框架 | **Vue 3** | ADR-2：学习曲线平缓，SFC 接近原生 HTML，零基础可快速上手 |
| UI 组件库 | **Element Plus** | ADR-2：表单、表格、分页、对话框等组件直接对应需求 |
| 状态管理 | **Pinia** | ADR-2：Vue 3 官方推荐 |
| HTTP 客户端 | **Axios** | P4 讲解文档：拦截器统一带 Token + 401 处理 |
| 后端框架 | **Spring Boot 3 + Java 17** | ADR-3：团队有 Java 基础，类型安全适合订单状态机 |
| ORM | **Spring Data JPA** | ADR-3：方法命名约定自动生成查询 |
| 数据库 | **MySQL 8.0** | P2 辩论结论：唯一课程中学过的数据库 |
| 认证 | **JWT** | P4 讲解文档：无状态认证，24h 过期 |

## 七、开发顺序与依赖关系

根据 P4 讲解手册建议的开发顺序，需求发布板块属于 **第 3 步**：

```
第 1 步：数据库建表 + 项目脚手架 + CI/CD
    ↓
第 2 步：用户注册 & 登录（先做完，这是所有功能的依赖）
    ↓
第 3 步：需求发布板块 ← 你在这里
    ├── 先做后端接口（POST/GET tasks）
    ├── 再做前端页面（发布页 + 列表页 + 详情页）
    └── 前后端联调
    ↓
第 4 步：接单 + 订单状态流转（依赖需求发布板块完成）
    ↓
第 5 步：评价 + 消息通知
    ↓
第 6 步：测试 + CI/CD + Bug 修复
```

**依赖说明：**
- 需求发布板块 **依赖** 用户认证模块（JWT 中间件 + Axios 拦截器必须先完成）
- 接单 & 订单模块 **依赖** 需求发布板块（订单需要关联 Task）

## 九、开发注意事项

### 前端注意（P4 给前端的建议）：

1. **先和后端对好 API 格式**（请求参数、响应字段、错误码），用 Postman / Apifox 先测通
2. **集中处理 Token** — Axios 拦截器一劳永逸，别在每个请求里手动加 header
3. **用户信息存 Pinia** — 用户角色(role)用于控制「发布需求」按钮的显隐
4. **表单校验** — 客户端做一遍即时反馈，但别信任它（后端必须再验一遍）

### 后端注意（P4 给后端的建议）：

1. **先建表** — 按 P3 的建表 SQL 执行，注意外键顺序（users → categories → tasks）
2. **统一错误处理** — 写一个 `GlobalExceptionHandler`，别在每个 Controller 里 try-catch
3. **接口先返回 mock 数据** — 不用等数据库全建好才开始写 Controller
4. **参数二次校验** — 前端校验不能替代后端校验

### AI 协作注意（来自 P1/P2 反思日志）：

1. **Prompt 要详细** — 提供完整的请求/响应 JSON 格式，让 AI 照格式生成，不要只说"写个 CRUD"
2. **不要让 AI 猜** — 提供团队实际技术栈和约束条件，避免 AI 输出公式化内容
3. **AI 输出必须人工审查** — AI 协作契约第一条：所有 AI 输出必须经过人工审查后才能合入项目
4. **换模型/清记录** — 如果 AI 输出重复趋于一致，换一个模型或清除之前的对话记录

## 十一、通用规范速查（来自 P3 API 规范文档）

**Base URL：** `https://api.campus-help.com/v1`（本地开发使用 `http://localhost:8080/v1`）

**认证方式：** 请求头 `Authorization: Bearer <token>`

**通用响应结构：**

成功：
```json
{ "code": 200, "message": "success", "data": { "..." }
```

失败：
```json
{ "code": 400, "message": "错误描述", "error": "详细错误信息" }
```

**常用错误码：**

| 错误码 | 说明 |
|:------:|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未认证/Token失效 |
| 403 | 无权限（如 provider 尝试发布需求） |
| 404 | 资源不存在（需求ID不存在） |
| 422 | 参数校验失败 |
| 429 | 请求过于频繁 |
| 500 | 服务器内部错误 |
