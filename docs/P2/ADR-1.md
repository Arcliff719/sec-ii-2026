
# ADR-1：架构风格选择分层架构和前后端分离

## 状态
已接受

## 背景

CampusHub 校园互助服务平台进入体系结构设计阶段，需要为 MVP 选定架构风格。项目面临以下约束：

- **团队**：4 人学生团队（1 测试 + 1 需求 + 1 开发 + 1 架构），成员仅有 Python/Java/C++ 课程基础，无实际项目开发经验
- **周期**：10 周，MVP 范围限定为用户管理、需求发布与浏览、订单流程管理
- **业务特征**：核心流程为"发布需求 → 接单 → 订单状态流转 → 双方评价 → 信用积分更新"，以请求-响应为主，无明显高并发场景
- **候选方案**：MVC、分层架构、微服务三种风格

候选架构的多维度对比（5 分制，1 为最优）：

| 对比维度 | MVC | 分层架构 | 微服务 |
|----------|-----|---------|--------|
| 开发复杂度 | 1 | 2 | 5 |
| 可维护性 | 2 | 4 | 3 |
| 可扩展性 | 1 | 2 | 5 |
| 性能 | 1 | 2 | 5 |
| 团队学习成本 | 1 | 2 | 5 |
| 10 周内可行性 | 1 | 2 | 5 |

## 决策

采用**分层架构（展示层 → 业务逻辑层 → 数据访问层 → 数据库）+ 前后端分离**的架构风格。

具体而言：
- 后端采用 Controller → Service → Repository → Model 的四层划分
- 前端为独立项目，通过 RESTful API 与后端通信
- 整体为单体部署，各层在同一个后端进程中运行

## 理由

1. **团队学习成本可控**：分层架构与课程所学的 MVC 模式相近，Controller/Service/DAO 的职责划分直观易懂，团队可在 1-2 周内上手
2. **前后端可并行开发**：前后端通过 API 契约（JSON over HTTP）解耦，架构负责人负责前端页面、开发负责人负责后端 API，4 人分工清晰不阻塞
3. **10 周可行性高**：前后端分离的模式部署简单，无需服务发现、分布式事务、容器编排等额外基础设施
4. **核心业务天然适配分层**：订单状态机、信用积分计算等核心逻辑适合在 Service 层集中实现，便于单元测试和维护
5. **可演进性**：分层边界清晰，未来若消息推送等模块需要独立扩展，可从 Service 层拆分为独立服务，架构升级成本可控
6. **为什么不选 MVC**：MVC虽简单已上手，擅长快速部署，但是面对本项目后续的扩展与测试难以展开，且一旦某部分出现漏洞将导致整个程序崩溃
7. **为什么不选微服务**：4 人团队 + 10 周周期下，微服务的基础设施搭建将消耗大量非业务时间，难以在规定时间实现部署，且团队学习成本过高

## 后果

- **正面影响**：
  - 前后端职责清晰，API 契约驱动开发，减少联调摩擦
  - 分层结构使单元测试目标明确（Repository 测数据访问、Service 测业务逻辑、Controller 测接口行为）
  - 单体部署运维简单，适合团队快速上手与快速完成项目
- **负面影响**：
  - 长期维护中若分层纪律松懈，可能出现跨层调用导致"大泥球"
  - 单体后端整体扩展，无法针对热点模块独立扩容（但 MVP 阶段无需担忧）
- **需要关注的风险**：
  - API 契约不清晰将导致前后端联调返工——需在前期完成接口文档定稿
  - 团队成员可能将业务逻辑泄漏到 Controller 层——需通过 Code Review 守住分层边界

## AI 辅助记录
- AI 初稿内容摘要：选择分层架构的基础上给出与讨论类似的理由，同时完善了Controller → Service → Repository → Model 的四层划分
- 人工修订内容：删除了部分AI额外考虑的内容，对AI的话术作精简，以及对于理由部分作更符合团队现实情况的修改。
- 修订理由：AI在部分层面考虑过多，例如部署成本问题，以及在理由方面部分情况考虑与团队实际有偏差，与辩论内容不符。
