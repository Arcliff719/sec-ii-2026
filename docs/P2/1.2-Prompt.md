## 2. 关键架构决策记录（ADR）

### ADR-001：使用单体后端而非微服务

- **背景**：MVP 仅支持核心流程，无独立高负载模块
- **决策**：采用单体后端（如 Django REST / Flask / NestJS）
- **理由**：4 人团队无需服务发现、分布式事务；单体部署简单
- **后果**：未来需重构拆分，但 MVP 阶段可接受

### ADR-002：订单状态机在应用层显式实现

- **背景**：订单状态流转需要严格校验
- **决策**：在业务逻辑层用代码实现状态转换校验
- **理由**：业务规则清晰，便于单元测试；避免数据库逻辑分散
- **后果**：需保证并发接单时的数据一致性（行锁或乐观锁）

### ADR-003：信用积分采用最终一致性更新

- **背景**：评价完成后需重新计算双方信用分
- **决策**：评价提交后立即发送异步任务更新积分
- **理由**：用户对实时性要求不高（<2秒可接受）；避免阻塞 HTTP 请求
- **后果**：需要任务队列与幂等处理，但 MVP 可简化

### ADR-004：不使用前端重型状态管理库

- **背景**：MVP 页面数 ≤ 8
- **决策**：使用框架自带响应式状态 + 服务端缓存
- **理由**：减少前端复杂性和学习成本
- **后果**：多标签页状态不同步，但影响小，可刷新页面解决

## 3. 风险识别与缓解策略

| 风险 ID | 风险描述 | 可能性 | 影响 | 缓解策略 |
|---------|----------|--------|------|-----------|
| R1 | 并发接单导致状态错误 | 中 | 高 | 数据库乐观锁；接单前再次校验状态 |
| R2 | 10 周交付压力 | 高 | 高 | MVP 功能冻结；每日站会 + 燃尽图；核心流程优先 |
| R3 | 评价与积分更新失败 | 中 | 中 | 异步重试（最多3次）；记录日志表支持手工补偿 |
| R4 | 安全漏洞 | 低 | 高 | RBAC；API 鉴权；敏感字段脱敏；不使用明文密码 |
| R5 | 订单状态难以追溯 | 中 | 中 | 新增状态变更表（order_log） |
| R6 | 数据库设计过早复杂 | 中 | 中 | 适度反范式；使用索引覆盖高频查询 |

## 4. 总结与建议

1. **坚决选择单体 + 事务脚本**：这是 10 周 4 人团队能成功交付的唯一务实路径

2. **优先级排序**：
    - 第 1–3 周：用户管理 + 需求发布浏览
    - 第 4–7 周：订单状态流转 + 评价（核心闭环）
    - 第 8–9 周：信用积分 + 异步处理 + 安全加固
    - 第 10 周：集成测试 + 部署 + 文档

3. **技术栈建议**：
    - 后端：Python Django + DRF 或 Node.js + Express
    - 数据库：PostgreSQL
    - 异步任务：Redis + Celery
    - 前端：Vue 3 + Pinia 或 React + Zustand

4. **非功能需求红线**：
    - 订单状态机代码测试覆盖 ≥ 90%
    - 并发接单场景需压测
    - 所有 API 输出结构化日志
