### 3.1 用户注册 & 登录（P0 | 前后端 | ⭐⭐）

**前端负责：**
- 注册表单：姓名、邮箱、密码、手机号、角色选择（requester/provider）
- 登录表单：邮箱 + 密码
- 登录成功后存储 JWT Token（localStorage / cookie）
- 401 时自动跳转登录页

**后端负责：**
- `POST /v1/auth/register` — 校验参数 → bcrypt 加密密码 → AES 加密手机号 → 写入 users 表 → 返回 userId
- `POST /v1/auth/login` — 查邮箱 → 验 bcrypt → 生成 JWT → 返回 token + 用户信息
- JWT 中间件（拦截所有非公开请求, 解析 token, 注入当前用户）
- 密码强度校验（8-32 位, 含大小写+数字）
- 邮箱格式校验（`.edu` 结尾）

**需要什么：**
- 后端：Spring Security / Passport.js, jjwt / jsonwebtoken, bcrypt 库, AES 加密工具
- 前端：表单组件, Axios 拦截器（自动带 token + 401 处理）, 路由守卫

**示例代码（后端 — 注册核心逻辑）：**

```java
// 伪代码：注册逻辑
public User register(RegisterRequest req) {
    // 1. 校验邮箱格式 & 唯一性
    if (!req.email.endsWith(".edu")) throw new BadRequest("仅支持学校邮箱");
    if (userRepo.existsByEmail(req.email)) throw new BadRequest("邮箱已被注册");
    
    // 2. 密码强度校验
    if (!req.password.matches("^(?=.*[a-z])(?=.*[A-Z])(?=.*\\d).{8,32}$"))
        throw new BadRequest("密码强度不足");
    
    // 3. bcrypt 哈希
    String hash = BCrypt.hashpw(req.password, BCrypt.gensalt(10));
    
    // 4. AES 加密手机号
    String encryptedPhone = AESUtil.encrypt(req.phone);
    
    // 5. 入库
    User user = new User();
    user.setName(req.name);
    user.setEmail(req.email);
    user.setPasswordHash(hash);
    user.setPhone(encryptedPhone);
    user.setRole(req.role);
    return userRepo.save(user);
}
```

**示例代码（前端 — Axios 拦截器）：**

```javascript

  const token = localStorage.getItem('token');
  if (token) config.headers.Authorization = `Bearer ${token}`;

  res => res,

  }
```

```java
public PageResult<TaskDTO> listTasks(int page, int size, 
        Integer categoryId, String status, String keyword, String sortBy) {
    
    Pageable pageable = PageRequest.of(page - 1, size, 
        Sort.by("desc".equals(order) ? Sort.Direction.DESC : Sort.Direction.ASC, sortBy));
    
    Specification<Task> spec = Specification
        .where(categoryId != null ? (root, q, cb) -> cb.equal(root.get("categoryId"), categoryId) : null)
        .and(status != null ? (root, q, cb) -> cb.equal(root.get("status"), status) : null)
        .and(keyword != null ? (root, q, cb) -> cb.like(root.get("title"), "%" + keyword + "%") : null);
    
    Page<Task> taskPage = taskRepo.findAll(spec, pageable);
    
    return new PageResult<>(taskPage.map(this::toDTO));
}
```

### 3.3 接单 & 订单状态流转（P0 | 前后端 | ⭐⭐⭐）

**这是 P0 中最重要的业务逻辑，难度最高。**

**状态流转图：**

```
  接单           开始执行         完成
accepted ──→ in_progress ──→ completed
   │                            │
   └──────── cancelled ─────────┘
         (任意中间状态可取消)
```

**前端负责：**
- 需求详情页的「接单」按钮（仅 provider 可见, 且不能接自己的需求）
- 订单状态展示（时间线组件展示状态变更记录）
- 「确认完成」「取消订单」等操作按钮
- 根据状态禁用/启用不同按钮

**后端负责：**
- `POST /v1/orders` — 接单：校验角色（仅 provider）→ 校验不能接自己的 → 检查是否已被接 → 创建 Order + 更新 Task 状态
- `PUT /v1/orders/{id}/status` — 状态变更（进行中 / 完成 / 取消）
- `GET /v1/orders/{id}` — 订单详情（含状态时间线）
- `GET /v1/orders?role=requester|provider` — 我的订单列表
- **并发控制**：防止同一 Task 被多人同时接单

**需要什么：**
- 后端：状态机实现（状态模式或枚举流转表）, 数据库事务, 乐观锁/悲观锁
- 前端：状态时间线 UI 组件, 按钮权限控制

**示例代码（后端 — 接单防并发 + 业务校验）：**

```java
@Transactional
public Order acceptOrder(Long taskId, Long providerId) {
    // 1. 悲观锁锁定 Task 行，防止并发
    Task task = taskRepo.findByIdWithLock(taskId)
        .orElseThrow(() -> new NotFound("需求不存在"));
    
    // 2. 不能接自己的需求
    if (task.getRequesterId().equals(providerId))
        throw new BusinessError("不能接自己发布的需求", 40005);
    
    // 3. 查该分类的交互模式（一人接单 vs 多人参与）
    Category cat = categoryRepo.findById(task.getCategoryId()).orElseThrow();
    
    // 4. 一人接单模式：检查是否已有活跃 Order
    if (isOneToOne(cat)) {
        boolean hasActive = orderRepo.existsByTaskIdAndStatusNot(taskId, "cancelled");
        if (hasActive) throw new BusinessError("该需求已被接单", 40004);
    }
    
    // 5. 创建订单 + 更新需求状态
    Order order = new Order();
    order.setTaskId(taskId);
    order.setProviderId(providerId);
    order.setRequesterId(task.getRequesterId());
    order.setStatus("accepted");
    order = orderRepo.save(order);
    
    task.setStatus("in_progress");
    taskRepo.save(task);
    
    // 6. 触发通知（见 3.5）
    notificationService.notify(task.getRequesterId(), 
        NotifyType.ORDER_ACCEPTED, task.getTitle());
    
    return order;
}
```

**状态流转伪代码：**

```java
// 状态变更
public void updateStatus(Long orderId, String newStatus, Long operatorId) {
    Order order = orderRepo.findById(orderId).orElseThrow();
    
    // 校验操作者是否为订单参与方
    if (!order.getRequesterId().equals(operatorId) 
        && !order.getProviderId().equals(operatorId))
        throw new Forbidden("无权操作");
    
    // 状态流转校验
    if (!isValidTransition(order.getStatus(), newStatus))
        throw new BusinessError("状态变更不合法");
    
    order.setStatus(newStatus);
    if ("completed".equals(newStatus)) order.setCompletedAt(now());
    if ("cancelled".equals(newStatus)) order.setCancelledAt(now());
    orderRepo.save(order);
}
```

### 3.4 评价 & 信用体系（P1 | 前后端 | ⭐⭐⭐）

**业务规则（来自 P3 设计）：**
- 评价是**单向**的：只有服务方（provider）可以评价需求方（requester）
- 评价前提：订单状态为 `completed` + 分类 `has_review = 1`
- 一个订单只能评价一次
- 信用分计算：需求方的信用分 = 所有被评分的平均值

**前端负责：**
- 订单完成后的「评价」按钮（仅 provider 可见，且仅 has_review 的分类显示）
- 评分星级组件 + 评价文本输入
- 用户资料页展示信用分

**后端负责：**
- `POST /v1/reviews` — 校验订单状态 → 校验分类 has_review → 校验角色 → 校验未重复 → 写入 reviews 表 → 重算信用分
- `GET /v1/users/{id}/reviews` — 查看某用户的评价记录
- 信用分计算：`SELECT AVG(score) FROM reviews r JOIN orders o ON r.order_id = o.id WHERE o.requester_id = ?`

**需要什么：**
- 前端：星级评分组件（可用第三方库如 vue-star-rating / react-rating）
- 后端：事务保证评价写入 + 信用分重算原子性

**示例代码（后端 — 评价提交 & 信用分更新）：**

```java
@Transactional
public Review submitReview(Long orderId, Long reviewerId, int score, String comment) {
    Order order = orderRepo.findById(orderId)
        .orElseThrow(() -> new NotFound("订单不存在"));
    
    // 1. 只有服务方可评价
    if (!order.getProviderId().equals(reviewerId))
        throw new Forbidden("只有服务方可评价");
    
    // 2. 订单必须已完成
    if (!"completed".equals(order.getStatus()))
        throw new BusinessError("订单未完成，不可评价", 40006);
    
    // 3. 分类必须支持评价
    Task task = taskRepo.findById(order.getTaskId()).orElseThrow();
    Category cat = categoryRepo.findById(task.getCategoryId()).orElseThrow();
    if (cat.getHasReview() != 1)
        throw new BusinessError("该分类不支持评价", 40006);
    
    // 4. 不能重复评价（DB 层 uk_order_id 兜底）
    if (reviewRepo.existsByOrderId(orderId))
        throw new BusinessError("已评价过该订单", 40007);
    
    // 5. 写入评价
    Review review = new Review();
    review.setOrderId(orderId);
    review.setScore(score);
    review.setComment(comment);
    review = reviewRepo.save(review);
    
    // 6. 重算需求方信用分
    Double avgScore = reviewRepo.calcAvgScoreByRequester(order.getRequesterId());
    userRepo.updateCreditScore(order.getRequesterId(), avgScore);
    
    return review;
}
```

### 3.5 消息通知（P1 | 前后端 | ⭐⭐⭐）

**触发时机（需要在相关业务代码中埋点）：**

| 触发事件 | 通知接收人 | 通知内容模板 |

| 有人接单 | 需求方 | "XXX 接了你的需求「YYY」" |

| 订单取消 | 需求方/服务方 | "订单「YYY」已被取消" |

**前端负责：**
- 导航栏通知图标 + 未读数量角标

**后端负责：**
- 通知生成：在各业务 Service 中调用 `notificationService.create(receiverId, type, title, content)`
- `GET /v1/notifications` — 当前用户通知列表（分页）
- `PUT /v1/notifications/{id}/read` — 标记已读
- `GET /v1/notifications/unread-count` — 未读数量

**需要什么：**
- 前端：未读角标组件, 轮询或 WebSocket（为简单起见，建议轮询 30s）

按依赖关系，建议的开发顺序：


## 六、给各角色的建议

### 如果你负责前端

1. **先和后端对好 API 格式**（请求参数、响应字段、错误码），用 Postman / Apifox 先测通
2. **集中处理 Token** — Axios 拦截器一劳永逸，别在每个请求里手动加 header
3. **状态管理** — 用户信息（id、role、creditScore）存 Pinia / Zustand，别到处传 props
4. **表单校验** — 客户端做一遍即时反馈，但别信任它（后端必须再验一遍）

### 如果你负责后端

1. **先建表** — 按 P3 的建表 SQL 执行，注意外键顺序
2. **统一错误处理** — 写一个 `GlobalExceptionHandler`，别在每个 Controller 里 try-catch
3. **事务管理** — 接单（Task + Order）、评价（Review + 信用分）必须在一个事务里
4. **接口先返回 mock 数据** — 不用等数据库全建好才开始写 Controller

### 如果你负责测试

1. 拿 P3 的错误码表，给每个错误码写一个测试用例
2. 核心流程的集成测试：注册 → 登录 → 发布 → 接单 → 完成 → 评价
3. 边界测试：空参数、超长字符串、负金额、非法状态转换
4. 重点关注 AI 生成的测试：检查是否有「断言了但实际没覆盖」的空洞测试

## 七、AI 协作关键提示

P3 阶段的 AI 协作经验可以直接复用到 P4：

| 场景 | 推荐 Prompt 策略 | 不推荐 |
|------|---------------|--------|
| 生成 CRUD 接口 | 提供完整的请求/响应 JSON 格式，让 AI 照格式生成 | 只说"写个 CRUD" |
| 复杂业务逻辑 | 先描述业务规则（状态流转图、校验条件），再让 AI 生成代码 | 把规则和代码需求混在一起 |
| 调试 | 提供报错信息 + 复现步骤 + 相关代码片段，明确问"根因是什么" | 只贴报错信息 |
| 测试生成 | 提供接口文档 + 所有错误码 + 边界条件清单 | 只给接口路径 |
